# Bug report: algorithmic uppercase commands fail in ltxpng

Date: 2026-07-15 21:17:39

## Summary

Rendering an `algorithm` / `algorithmic` fragment that uses old uppercase `algorithmic` commands such as `\REQUIRE`, `\ENSURE`, `\STATE`, `\FOR`, `\ENDFOR`, `\IF`, `\ELSE`, `\ENDIF`, and `\RETURN` fails with:

```text
! Undefined control sequence.
l.9 \REQUIRE
```

The immediate cause is a package/syntax mismatch in auto-detection: `ltxpng` detects `\begin{algorithmic}` and auto-loads `algpseudocode`, but the input uses the command syntax from the older `algorithmic` package.

## Reproduction

Example input:

```sh
echo '\begin{algorithm}
\caption{Hardware-Aware Prefix Scheduler}
\begin{algorithmic}[1]
\REQUIRE Active requests $r \in \{1, \ldots, R\}$
\ENSURE Selected per-request prefix lengths $\ell_1^*, \ldots, \ell_R^*$
\STATE Initialize states
\RETURN $(\ell_1^*, \ldots, \ell_R^*)$
\end{algorithmic}
\end{algorithm}' | ltxpng
```

Observed failure:

```text
! Undefined control sequence.
l.9 \REQUIRE
```

## Diagnosis

There are two commonly suspected causes, but only one explains the observed failure directly.

### 1. Display-math wrapping is not the direct cause in the current code

A possible diagnosis is that `ltxpng` always wraps input in display math:

```tex
\[
\begin{algorithm}
...
\end{algorithm}
\]
```

If that happened, it would indeed be invalid: `algorithm` is a text-mode float and cannot live inside display math.

However, the current implementation does not always wrap in `\[...\]`. In `src/detect.c`, fragments beginning with `\begin{...}` are classified as `FRAG_RAW`, and in `src/template.c`, raw fragments are emitted directly without display-math wrapping.

Therefore, for input beginning with `\begin{algorithm}`, the current code should use raw mode automatically. The observed error at `\REQUIRE` also indicates LaTeX got as far as the `algorithmic` environment; if the fragment had been wrapped in display math, LaTeX would typically fail earlier around `\begin{algorithm}`.

That said, `--raw` remains a useful workaround if auto-classification fails for a differently shaped input.

### 2. Package/syntax mismatch is the immediate root cause

`src/detect.c` currently contains:

```c
static const pkg_entry pkg_table[] = {
  {"\\begin{algorithmic}", "algpseudocode"},
  {"\\State",              "algpseudocode"},
  {"\\begin{algorithm}",   "algorithm"},
  {"\\begin{tikzpicture}", "tikz"},
};
```

So `\begin{algorithmic}` causes `ltxpng` to load:

```tex
\usepackage{algpseudocode}
```

But `algpseudocode` uses CamelCase commands:

```tex
\Require
\Ensure
\State
\For
\EndFor
\If
\Else
\EndIf
\Return
```

The user's fragment uses old `algorithmic` package commands:

```tex
\REQUIRE
\ENSURE
\STATE
\FOR
\ENDFOR
\IF
\ELSE
\ENDIF
\RETURN
```

Because `algpseudocode` does not define those uppercase commands, LaTeX reports:

```text
! Undefined control sequence.
l.9 \REQUIRE
```

## Additional issue: `algorithm` is a float

Even if the commands are changed from uppercase to CamelCase, `\begin{algorithm}` can still be problematic in the current fragment wrapper because `algorithm` is a float. Floats are not always valid inside the `standalone` / `varwidth` context used by `ltxpng` for fragments.

For fragment rendering, a safer pattern is to render the non-floating `algorithmic` block directly, optionally using `\captionof{algorithm}` from the `caption` package instead of the `algorithm` float.

## Workarounds

### Workaround A: Use `algpseudocode` syntax and avoid the float

```sh
cat <<'EOF' | ltxpng --raw \
  -p '\usepackage{algorithm}' \
  -p '\usepackage{caption}' \
  -o algo.png
\captionof{algorithm}{Hardware-Aware Prefix Scheduler}
\begin{algorithmic}[1]
\Require Active requests $r \in \{1, \ldots, R\}$
\Ensure Selected per-request prefix lengths $\ell_1^*, \ldots, \ell_R^*$
\State Initialize states
\Return $(\ell_1^*, \ldots, \ell_R^*)$
\end{algorithmic}
EOF
```

### Workaround B: Use full-document mode with the old `algorithmic` package

If the user wants to keep uppercase commands, provide a full LaTeX document. `ltxpng` detects `\documentclass` and compiles the document as-is, then crops it.

```sh
cat <<'EOF' | ltxpng -o algo.png
\documentclass{article}
\usepackage{amsmath,amssymb}
\usepackage{algorithm}
\usepackage{algorithmic}
\pagestyle{empty}
\begin{document}

\begin{algorithm}
\caption{Hardware-Aware Prefix Scheduler}
\begin{algorithmic}[1]
\REQUIRE Active requests $r \in \{1, \ldots, R\}$
\ENSURE Selected per-request prefix lengths $\ell_1^*, \ldots, \ell_R^*$
\STATE Initialize states
\RETURN $(\ell_1^*, \ldots, \ell_R^*)$
\end{algorithmic}
\end{algorithm}

\end{document}
EOF
```

## Would flags solve this?

Partially, but not cleanly for this exact case.

- `--raw` avoids display-math wrapping, but it does not define `\REQUIRE`.
- `-p` can add packages, but adding `\usepackage{algorithmic}` to a fragment may conflict with the auto-loaded `algpseudocode`, because `ltxpng` already auto-detects `\begin{algorithmic}` and loads `algpseudocode`.
- Full-document input works because package selection is then controlled entirely by the user.

Therefore, flags are useful, but the robust solution is to update package detection and/or provide stronger override controls.

## Recommended fixes

1. Detect old uppercase `algorithmic` commands.

   If any old-style commands are present, load `algorithmic` instead of `algpseudocode`:

   ```tex
   \REQUIRE
   \ENSURE
   \STATE
   \FOR
   \ENDFOR
   \IF
   \ELSE
   \ENDIF
   \RETURN
   ```

2. Keep loading `algpseudocode` for CamelCase commands:

   ```tex
   \Require
   \Ensure
   \State
   \For
   \EndFor
   \If
   \Else
   \EndIf
   \Return
   ```

3. Add an override such as `--no-auto-packages` or explicit package-selection controls, so users can avoid auto-package conflicts.

4. Consider special handling for `\begin{algorithm}` floats. Possible options:

   - document that `algorithm` floats should be supplied as full documents;
   - rewrite/guide users toward `\captionof{algorithm}` plus `algorithmic` for fragment mode;
   - add a float-safe rendering mode that uses a normal document class plus cropping instead of `standalone[varwidth]` for such fragments.

## Verdict

The reported failure is primarily caused by package/syntax mismatch: `ltxpng` loads `algpseudocode`, while the input uses old uppercase `algorithmic` commands.

The display-math wrapping explanation is not accurate for current `ltxpng` behavior when the input starts with `\begin{algorithm}`, because that input is classified as raw. However, `algorithm` floats are still a separate compatibility concern in fragment mode.
